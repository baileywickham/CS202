Lab 8, CSC 203 - depth first search

This lab explores a recursive algorighm: a depth-first search on a two-dimensional grid. This depth-first search is a simple "pathing" algorithm, in that it can be used to calculate a path from one point on the grid to another, while avoiding obstacles.

Objectives

  • To implement depth first search
  • Think about trajectory planning on a 2D grid structure
  • Create a test framework that can be used for the class project, when it addresses pathing.

Getting started - Background

This program uses the same Spritely library as the class project. It takes advantage of the fact that Spritely allows you display graphics whenever you want in your code, rather than having a main animation loop as is more typical with graphics programming. Because of this, a recursive algorithm can be implemented directly, using recursive method calls, and still use animation to show you its progress.

Open the provided Java file and read through the code. You will adapt this code to the project in an assignment. The code creates a grid data structure, where it stores the state of each square. It then creates a SpriteWindow from the spritely library; this window does the actual display. Then, in the run() method, it calls searchForPath() to look for the water. The current version only looks to the right, and gives up if it hits an obstacle.

Main also has an option to run unit tests. You'll see that the unit tests create simpler words, and apply the same algorithm. You should test your algorithm on a variety of layouts that are more challenging than the one provided. You might need to change existing tests, once your algorithm does a better job of finding the water.

This program uses both the Spritey and Testy libraries.

Step 1

Implement a depth first search. You can visit the neighboring nodes in whichever order you like. Perhaps you can try visiting in various orders, and ultimately select an ordering that clearly illustrates that the search runs into the obstacles at some point. In other words, don't pick directions that result in never visiting a grid cell that is occupied by an obstacle.

Step 2

After the search completes, make sure that the code in place to display the path. Be sure to think about how cells are being added to path, and how the path is represented. In the below example image with a slightly modified layout, all the cells with a yellow square have been visited but are not a part of the current path, and the grid cells that make up the current path are shown with red. You have been provided with images for both, in the images directory.

For avoidance of doubt, when you are finished, the wyverin should always find the water with the default layout. Make sure that you display a frame of animation for every change to the grid. That is, don't collapse more than one change into a single animation frame.
